# Decisions

Append-only log of architecture and collaboration decisions, shared between both founders. Newest at the bottom. Record the *why*, not just the *what*. Product and strategy decisions live in the Google Doc; this file is for how we build and work.

## 2026-06-19

- **`AGENTS.md` is the single source of truth for agents.** Codex reads `AGENTS.md` natively and `CLAUDE.md` only imports it, so both agents and both founders share one contract instead of two that drift. *Why:* we work with multiple agents (Claude Code, Codex) and a second founder; divergent instructions are how agents end up working blind.
- **Keep Aymane's scaffold as the technical base; rebuild the core mechanic on top.** The scaffold (signup, global directory, open DM) is kept for its working Supabase wiring, but the discreet-like + match-gate mechanic is built over it rather than starting from scratch. *Why:* the wiring is reusable; the open-DM flow violates the discreet double opt-in invariant and cannot ship as is.
- **Phase 1 build order: Bloc 0 (foundations), then 2 (like/match), then 1 (presence), then 3 (chat gate), then 4 (safety).** *Why:* the like/match mechanic is the hypothesis the whole product rests on, so it comes right after the data model it depends on.
- **Repo language is English, conversation can be French.** Code, comments, docs and commits are English; talking to agents in French is fine. *Why:* durable artifacts must be readable by both founders and any future contributor.
- **`docs/` holds the shared roadmap and decision log.** `docs/roadmap.md` for status and plan, `docs/decisions.md` for decisions. *Why:* Todoist is personal to Marwane and Aymane cannot see it; the shared trace must live in the repo.
- **V1 auth: Supabase anonymous sign-in, no account creation at the door.** Scanning the QR triggers `signInAnonymously()` automatically, so every visitor gets a real `auth.users` row (UUID + JWT session) with zero friction — no email, no SMS, no password. *Why:* the product is scan → see who's here, right now; a signup wall at a noisy bar kills the funnel. An anonymous session still gives us `auth.uid()`, so RLS is fully enforceable from day one (security-first) — which "no auth at all" cannot do.
- **Identity model: ephemeral presence and matches, persistent profile.** What persists is the profile (so a regular doesn't rebuild it every night). What is ephemeral is presence (fades on leave/timeout) **and the match/chat** (dies with the night). *Why:* the ephemerality of the match is the forcing function — you matched, you have *tonight* to go talk IRL. Persisting matches would reintroduce the async pen-pal dynamic the north star rejects (in-person first). We never promise users they can "retrieve their matches later".
- **Profile = first name + photo (required) + bio (optional). Email is optional and collected later, never at entry.** The required photo + first name is the soft identity / women-first layer (you see a real face), not a verified email. Email is offered after onboarding, framed as "get notified about the next live night" — it triggers the anonymous→permanent account upgrade (same UUID, profile becomes cross-device recoverable). *Why:* email earns us a re-engagement channel and the only real returning-user retention metric, but framed around bringing people back to a *live* room, not maintaining async chats. Email is PII: it must never be exposed to other users (RLS, no `select("*")`), and storing it touches the open RGPD / Supabase-region (EU vs US) question to settle with Aymane before any public test.
- **PII lives in a separate `profile_private` table, not in `profiles`.** `profiles` holds only public columns (first name, photo, bio, gender, interested_in); email/phone go in `profile_private` with owner-only RLS on every operation. *Why:* RLS is row-level and cannot hide a column of a row that is otherwise readable. Isolating PII in its own table is the only hard guarantee email never leaks — it does not depend on every future query remembering to avoid `select("*")`.
- **Discreet like enforced in RLS, not just the UI.** `likes` SELECT is `liker_id = auth.uid()`: you can read only the likes you sent, never one targeting you. The only signal that surfaces is a `matches` row, and only when mutual. *Why:* invariant 3 (no one is ever told they were liked) must hold at the database, so a bug or a direct API call can't leak "who liked me".
- **Matches are created by a `SECURITY DEFINER` trigger on `likes`, never by users.** `handle_new_like()` checks for the reciprocal like in the same venue and inserts the ordered pair into `matches`; EXECUTE is revoked from the API roles. *Why:* the reciprocity check must read `likes` rows the user is not allowed to SELECT, so it runs as definer; keeping inserts out of users' hands means the match is the single trustworthy mutual signal.
- **Messages are tied to a `match_id` (no open DM possible).** A message references a match and RLS requires the sender to be one of its two members. *Why:* makes the "no open messaging" red line structural — a message literally cannot exist without a prior mutual match.
- **Profile carries `gender` + `interested_in` (set of woman/man/nonbinary) from the start.** *Why:* `interested_in` is meaningless without the other person's `gender` to filter on, so both ship together rather than bolting gender on later. (`interested_in` is an array to allow more than one.)
- **Likes and matches are venue-scoped; ephemerality (expiry/cleanup) is deferred to Bloc 1/3.** Foundations ship the tables with no TTL yet; the "match dies with the night" cleanup job is a later bloc (noted in roadmap so it isn't forgotten). *Why:* keep Bloc 0 to the data model; the forcing-function expiry needs the presence timeout logic that lands with Bloc 1.
- **`NEXT_PUBLIC_SUPABASE_ANON_KEY` renamed to `NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY`.** The client now uses the new `sb_publishable_...` key, and `lib/supabase.ts` is typed with the generated `Database` type (`lib/database.types.ts`). *Why:* Supabase's new publishable key is the recommended browser key; the var name now matches reality.
- **i18n: code stays English always; user-facing UI text is bilingual FR/EN via a lightweight dictionary, no framework yet.** Code, identifiers, comments, commits remain English (unchanged convention). For the *displayed* strings, we will centralize them in a single dictionary (e.g. `lib/strings.ts`, `{ fr, en }`) and pick the locale from the venue (`venues.city`: Paris → fr, NYC → en), with browser language as the fallback on pre-venue pages (landing, profile). No `next-intl` / locale routing / message catalogs until the surface actually justifies it (plurals, localized dates, many screens, external translation). *Why:* the Paris and NYC test crowds are French- and English-speaking from the very first night, but the whole app is ~15 strings — a full i18n framework would be premature over-engineering; centralizing strings now is the cheap insurance that keeps a later switch to one file, not a hunt across components. Not built yet — planned as a small task at the start of Bloc 1 (see roadmap).
