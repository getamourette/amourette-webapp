# Decisions

Append-only log of architecture and collaboration decisions, shared between both founders. Newest at the bottom. Record the *why*, not just the *what*. Product and strategy decisions live in the Google Doc; this file is for how we build and work.

## 2026-06-19

- **`AGENTS.md` is the single source of truth for agents.** Codex reads `AGENTS.md` natively and `CLAUDE.md` only imports it, so both agents and both founders share one contract instead of two that drift. *Why:* we work with multiple agents (Claude Code, Codex) and a second founder; divergent instructions are how agents end up working blind.
- **Keep Aymane's scaffold as the technical base; rebuild the core mechanic on top.** The scaffold (signup, global directory, open DM) is kept for its working Supabase wiring, but the discreet-like + match-gate mechanic is built over it rather than starting from scratch. *Why:* the wiring is reusable; the open-DM flow violates the discreet double opt-in invariant and cannot ship as is.
- **Phase 1 build order: Bloc 0 (foundations), then 2 (like/match), then 1 (presence), then 3 (chat gate), then 4 (safety).** *Why:* the like/match mechanic is the hypothesis the whole product rests on, so it comes right after the data model it depends on.
- **Repo language is English, conversation can be French.** Code, comments, docs and commits are English; talking to agents in French is fine. *Why:* durable artifacts must be readable by both founders and any future contributor.
- **`docs/` holds the shared roadmap and decision log.** `docs/roadmap.md` for status and plan, `docs/decisions.md` for decisions. *Why:* Todoist is personal to Marwane and Aymane cannot see it; the shared trace must live in the repo.
- **V1 auth: Supabase anonymous sign-in, no account creation at the door.** Scanning the QR triggers `signInAnonymously()` automatically, so every visitor gets a real `auth.users` row (UUID + JWT session) with zero friction — no email, no SMS, no password. *Why:* the product is scan → see who's here, right now; a signup wall at a noisy bar kills the funnel. An anonymous session still gives us `auth.uid()`, so RLS is fully enforceable from day one (security-first) — which "no auth at all" cannot do.
- **Identity model: ephemeral presence and matches, persistent profile.** What persists is the profile (so a regular doesn't rebuild it every night). What is ephemeral is presence (fades on leave/timeout) **and the match/chat** (dies with the night). *Why:* the ephemerality of the match is the forcing function — you matched, you have *tonight* to go talk IRL. Persisting matches would reintroduce the async pen-pal dynamic the north star rejects (in-person first). We never promise users they can "retrieve their matches later".
- **Profile = first name + photo (required) + bio (optional). Email is optional and collected later, never at entry.** The required photo + first name is the soft identity / women-first layer (you see a real face), not a verified email. Email is offered after onboarding, framed as "get notified about the next live night" — it triggers the anonymous→permanent account upgrade (same UUID, profile becomes cross-device recoverable). *Why:* email earns us a re-engagement channel and the only real returning-user retention metric, but framed around bringing people back to a *live* room, not maintaining async chats. Email is PII: it must never be exposed to other users (RLS, no `select("*")`), and storing it touches the open RGPD / Supabase-region (EU vs US) question to settle with Aymane before any public test.
- **PII lives in a separate `profile_private` table, not in `profiles`.** `profiles` holds only public columns (first name, photo, bio, gender, interested_in); email/phone go in `profile_private` with owner-only RLS on every operation. *Why:* RLS is row-level and cannot hide a column of a row that is otherwise readable. Isolating PII in its own table is the only hard guarantee email never leaks — it does not depend on every future query remembering to avoid `select("*")`.
- **Discreet like enforced in RLS, not just the UI.** `likes` SELECT is `liker_id = auth.uid()`: you can read only the likes you sent, never one targeting you. The only signal that surfaces is a `matches` row, and only when mutual. *Why:* invariant 3 (no one is ever told they were liked) must hold at the database, so a bug or a direct API call can't leak "who liked me".
- **Matches are created by a `SECURITY DEFINER` trigger on `likes`, never by users.** `handle_new_like()` checks for the reciprocal like in the same venue and inserts the ordered pair into `matches`; EXECUTE is revoked from the API roles. *Why:* the reciprocity check must read `likes` rows the user is not allowed to SELECT, so it runs as definer; keeping inserts out of users' hands means the match is the single trustworthy mutual signal.
- **Messages are tied to a `match_id` (no open DM possible).** A message references a match and RLS requires the sender to be one of its two members. *Why:* makes the "no open messaging" red line structural — a message literally cannot exist without a prior mutual match.
- **Profile carries `gender` + `interested_in` (set of woman/man/nonbinary) from the start.** *Why:* `interested_in` is meaningless without the other person's `gender` to filter on, so both ship together rather than bolting gender on later. (`interested_in` is an array to allow more than one.)
- **Likes and matches are venue-scoped; ephemerality (expiry/cleanup) is deferred to Bloc 1/3.** Foundations ship the tables with no TTL yet; the "match dies with the night" cleanup job is a later bloc (noted in roadmap so it isn't forgotten). *Why:* keep Bloc 0 to the data model; the forcing-function expiry needs the presence timeout logic that lands with Bloc 1.
- **`NEXT_PUBLIC_SUPABASE_ANON_KEY` renamed to `NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY`.** The client now uses the new `sb_publishable_...` key, and `lib/supabase.ts` is typed with the generated `Database` type (`lib/database.types.ts`). *Why:* Supabase's new publishable key is the recommended browser key; the var name now matches reality.
- **Presence model: "the room of tonight", not a minute-by-minute timeout.** You are present from check-in until you explicitly leave or the night rolls over at a fixed local hour. There is **no short heartbeat timeout that hides you**. *Why:* a heartbeat only fires while the tab is open and active; at a bar the phone is in a pocket, screen locked. A short timeout would drop people who are physically in the room, and we cannot distinguish "phone in pocket, still here" from "went home" — both just stop heart-beating. A short window also empties the room and kills the cold-start. The "live" invariant is honoured at the granularity of *tonight*, which is the right grain for a bar. Risk accepted: you might match with someone who already left — but the match is ephemeral and the promise is "they're here tonight", same as real life.
- **The night ends at 06:00 local time, enforced by a `pg_cron` job (`bartap-close-ended-nights`, every 15 min).** `close_ended_nights()` sets `left_at` on every active presence past the first 06:00 (in the venue's timezone) strictly after its check-in — handles after-midnight check-ins (02:00 belongs to the night that ends that same morning). Venues carry an explicit `timezone` column (Paris → `Europe/Paris`, NYC → `America/New_York`) so the rollover does not hardcode a city map. *Why:* this cron, not a client timeout, is what empties the room for good; 06:00 maps to a bar closing and is predictable.
- **`check_in(venue)` RPC is the single check-in path: one room at a time, idempotent, also the heartbeat.** It leaves any active presence in a *different* venue, then bumps `last_seen_at` if already here or inserts a fresh row. `SECURITY INVOKER` so it runs under the caller's own RLS. *Why:* a clean atomic upsert against the partial unique index (`one active presence per profile`) without client-side juggling; reusing it as the heartbeat keeps one code path.
- **Explicit "leave the room" control ships in Bloc 1, not deferred to Bloc 4.** A button sets `left_at` and removes you from the room immediately. *Why:* it is cheap, it is the natural complement to the night-session model (the only way to disappear before the rollover), and it is a first concrete women-first control over your own visibility.
- **`profiles` and `presence` SELECT tightened to co-present users now that presence exists.** `profiles` is readable only for yourself, anyone co-present in a venue you are currently in, or anyone you have matched with (so a match stays readable after they leave); `presence` only for your own rows and the venues you are in. This closes the intentional Bloc 0 hole (profiles readable by any authenticated session). *Why:* the live invariant — you only see who is in the room you are standing in. The "or matched" clause keeps the match reveal working across a refresh and is forward-compatible with Bloc 3 chat.
- **RLS visibility helpers live in a `private` schema, not `public`.** An RLS policy that references the table it protects causes Postgres "infinite recursion in policy", so the visible-id sets are computed in `SECURITY DEFINER` helpers (`my_active_venue_ids`, `visible_profile_ids`) that bypass RLS. They are in a `private` schema PostgREST does not expose, so they are not reachable as `/rest/v1/rpc` endpoints. *Why:* break the recursion safely while keeping the callable API surface minimal (women-first: smallest possible attack surface), even though the data they return is non-sensitive.
- **i18n: code stays English always; user-facing UI text is bilingual FR/EN via a lightweight dictionary, no framework yet.** Code, identifiers, comments, commits remain English (unchanged convention). For the *displayed* strings, we will centralize them in a single dictionary (e.g. `lib/strings.ts`, `{ fr, en }`) and pick the locale from the venue (`venues.city`: Paris → fr, NYC → en), with browser language as the fallback on pre-venue pages (landing, profile). No `next-intl` / locale routing / message catalogs until the surface actually justifies it (plurals, localized dates, many screens, external translation). *Why:* the Paris and NYC test crowds are French- and English-speaking from the very first night, but the whole app is ~15 strings — a full i18n framework would be premature over-engineering; centralizing strings now is the cheap insurance that keeps a later switch to one file, not a hunt across components. Not built yet — planned as a small task at the start of Bloc 1 (see roadmap).
- **Bloc 3 chat is addressed by `match_id`, never by recipient profile id, and expires with the venue night.** The room links to `/chat/[matchId]` only after a reciprocal match exists; messages still reference `match_id`, and RLS requires the sender/reader to be one of the match members. `matches.expires_at` is computed from the venue timezone and the 06:00 local night boundary, and the existing `bartap-close-ended-nights` cron deletes expired matches, cascading messages. *Why:* this keeps "no open DM" structural at both URL and database levels, while making the match/chat disappear with the night so Paramour remains an in-person trigger, not an async inbox.
- **Bloc 4 18+ gate is a mandatory checkbox, stored owner-only in `profile_private`.** We do not collect birthdate and do not use external verification in v1. *Why:* the first real-night test needs a clear adult-only gate without adding sensitive date-of-birth data or a heavy vendor flow before the core spark is proven.
- **Invisible mode is a balanced visibility pause, not silent browsing.** `presence.is_visible=false` keeps the person checked in, hides them from other room grids, and pauses browsing/liking until they become visible again; existing matches/chats remain available. *Why:* women need a fast way to regain control of exposure without leaving the venue, but letting invisible users keep browsing would create an asymmetric, creepy dynamic.
- **Block is strong: hide both ways and close active interactions.** A `blocks` row excludes both profiles from each other's visible sets, prevents new likes/messages, and a trigger deletes existing likes and matches between the pair, cascading messages. *Why:* block is a safety boundary, not a preference filter. Keeping an existing chat alive after a block would undermine the control users expect.
- **Reports are actionable but moderation-light in v1.** A report stores a fixed reason plus an optional note, is readable only by the reporter via RLS, and the UI offers block immediately after reporting. There is no admin console yet. *Why:* v1 needs structured incident data and an immediate self-protection path, while full moderation tooling can wait until there is real usage to moderate.
- **Likes are venue-night scoped, not venue-scoped forever.** Likes carry the same 06:00-local `expires_at` boundary as matches, reciprocal matching requires both likes to share that boundary, and cleanup deletes expired likes with expired matches. *Why:* a like from a previous night should never create tonight's match or keep a person stuck in a stale "Liked" state. This preserves the live, in-person promise: profile persists, but presence, likes, matches, and chats are about tonight.
- **Profile-photo quality has a free baseline now and optional paid AI review later.** The client rejects unsupported/oversized files by default; `PROFILE_PHOTO_REVIEW_ENABLED=true` turns on a server-side vision review before upload, requiring a clear real photo centered on one visible adult human face. *Why:* the required photo is the soft-identity layer that makes a live dating room feel safer, but paid review should not run during routine local development before real venue testing. This is an MVP gate, not final abuse-proofing: the later hardening step is to move Storage upload and profile insert fully server-side so the review cannot be bypassed with direct Supabase calls.

## 2026-06-22

- **Git workflow is issue → branch → PR → squash-merge → delete branch, off `main`.** Branches are `feature/...`/`fix/...` cut from an up-to-date `main`, one branch per PR, reviewed and squash-merged so `main` stays one commit per PR. Issues are optional for trivial work, expected for anything worth tracking. *Why:* it is the GitHub-PR flow we already follow loosely; writing it down makes it the shared default for both founders and any agent, and squash-merge keeps `main` readable. The real failure mode to avoid is letting PRs pile up — a stack of long-lived branches against a moving `main` invites conflicts and DB drift, so open PRs get merged quickly.
- **Schema changes on the single shared remote DB: additive applied freely to test, behavioral/destructive coordinated.** One remote Supabase project is shared by both founders with no local stack, so testing schema-dependent code before a PR means applying the migration to that shared DB. While pre-launch (the shared DB is effectively the dev DB, no real users to protect), an additive migration may be applied early to test; a behavioral or destructive one is announced to the other founder before applying and the PR is merged promptly to resync code and DB. The migration file always ships in the PR. When real users exist, we split a dedicated prod project from this dev one and gate prod migrations to merge time. *Why:* a strict "apply only at merge" rule makes schema-dependent code untestable before the PR on a single shared DB; the additive-vs-behavioral split is what lets us keep testing without breaking the other founder's in-progress `main`. Protecting the shared DB like a prod is premature while there are no users — simplicity first, harden when launch makes it real.
- **Agents do not merge PRs or apply migrations to the shared DB autonomously.** An agent may write code and migration files and open the PR; merging and applying to the remote stay founder-gated. *Why:* both actions are shared and hard to reverse against a resource the other founder depends on, so a human stays in the loop even though agents do the rest of the work.

## 2026-06-28

- **The product name is Paramour.** User-facing app copy and durable product docs use Paramour as the brand name. Existing database object names that include the previous codename remain unchanged unless a migration explicitly renames them. *Why:* the public product needs one consistent name, while already-applied database identifiers should not be rewritten casually.
- **Language is user-selectable across EN/FR/ES.** Venue city and browser language remain the default, but an explicit language choice is stored locally and overrides those defaults across landing, profile, room, and chat. *Why:* test crowds may not match the venue default, and the control should be immediate without account settings or a heavier i18n framework.

## 2026-07-01

- **An admin dashboard is built at `/admin` inside the existing app, not as a separate site.** It is one more route in the same Next.js app / same domain, not a separate deployment or subdomain. *Why:* at this scale a second app is pure maintenance cost for no benefit; a route reuses the existing stack, and access is gated in-app plus by RLS. A subdomain / standalone admin app is a Phase 3 concern (venue dashboard) if ever.
- **Admin access is founder-only via an `admins` allow-list table + RLS, gated by real email/password login — never the service-role in the browser.** A small `public.admins(user_id)` table lists the founders; RLS policies grant admins read access to `reports`/`blocks`/aggregate stats and write access to `venues`. The two founder accounts sign in with Supabase email/password (email provider enabled, accounts created manually via MCP), separate from the public anonymous flow. The service-role key must never reach client code. *Why:* the DB treats every session identically, and an anonymous session cannot see other users' reports — so the DB must recognise the founders as privileged to make the dashboard show anything. An allow-list + RLS stays inside the security model already built (RLS is the real rampart), whereas a server-only service-role dashboard bypasses every RLS guarantee and is dangerous to hand-roll. A custom Postgres role was rejected as over-engineering for two people.
- **The admin auth mechanism is NOT generalised to public users in v1.** Only the two founder accounts get email login; regular visitors stay purely anonymous. *Why:* no premature abstraction — the same Supabase email-auth building block is reused for real user accounts if/when Phase 2 (retention, persistent accounts) needs them, but building a public login now is scope with no v1 payoff.
- **No public user login / profile recovery in v1; optional non-blocking email capture only.** Regular users stay anonymous. Everything valuable (presence, likes, matches, chats) is ephemeral and dies at 06:00; the only persistent thing is the profile (first name + photo + bio), which is cheap to recreate. So cross-device profile *recovery* (a real login flow) is deferred to Phase 2. What may ship in v1 is an *optional, never-blocking* email field framed as "get notified about the next live night", which attaches the email to the same anonymous UUID (and incidentally prepares later recovery). *Why:* the founding principle is zero signup friction at a noisy bar; a login flow solves a 20-second problem (rebuilding a cheap profile) while adding surface and friction. Email capture is a retention/growth hook the roadmap already wants, and is a different thing from login.
- **v1 identity persistence is per-device, and that is accepted.** The anonymous session lives in the browser's localStorage: a returning user on the same phone/browser is recognised automatically (no login), but clearing storage / switching device / private mode orphans the old profile and starts a fresh anonymous user. *Why:* acceptable for v1 because the profile is cheap to recreate and everything else is ephemeral by design. True per-person persistence is what the deferred Phase 2 accounts provide.
- **The v1 landing (`/`) is a gate page with three states, not a login page.** (1) New visitor via typed URL → short pitch + "scan the QR at the bar to join tonight" (no venue is reachable without scanning a QR). (2) Returning visitor with a live session + existing profile → their profile summary + "scan to check in", plus a direct link to any still-active match/chat from tonight. (3) After the night (presence/matches cleaned by the cron) → same as (2), "scan to join tonight". *Why:* in v1 there is no app and no scheduled-soirée concept; the only entry into a venue is scanning the physical QR on the spot. "Reconnecting" is just reopening the app with the stored session, not authenticating.
- **Venues get an `is_live` flag; `check_in()` refuses when a venue is not live.** New `venues.is_live boolean not null default false`. `check_in()` raises if the target venue is not live, closing the "reconnect any afternoon with an old QR link" hole (today `check_in()` checks neither time nor venue state). The admin dashboard is the start/stop switch: *start* sets `is_live=true`; *stop* sets `is_live=false` **and** closes all open presence in that venue (clean room emptying). The 06:00 `close_ended_nights()` cron stays the safety net and additionally flips `is_live=false` for venues whose night has ended, so a forgotten venue self-closes. Dev seed venues (`paris-test`, `nyc-test`) ship `is_live=true` so the dev flow keeps working. *Why:* liveness was purely time+presence with no way to prevent off-hours entry; `is_live` gates *entry* while the cron guarantees *cleanup* — two mechanisms, two distinct roles. This is a **behavioral** migration (it changes `check_in`), so per the shared-DB rule it must be announced to the other founder before applying and the PR merged promptly to resync code and DB.
- **Admin analytics are aggregate-only; no free-browsing of who liked/matched whom.** The dashboard shows per-night/per-venue aggregates (check-ins, likes sent, matches, chats started). "Who liked whom" is never exposed to anyone, including founders (it would break invariant 3 at its root). "Who matched whom" in nominative form is only ever surfaced inside the moderation context of a specific report, never as a browsable list. *Why:* a founder backdoor onto likes/matches destroys the discreet double opt-in trust model the instant it is discovered or leaks; aggregates prove the spark (the Phase 1 goal) without de-anonymising anyone.
- **The admin dashboard's first job is moderation, not analytics.** Reports currently land nowhere (readable only by service-role, no console), so the highest-value surface is a reports/blocks queue the founders can actually see and act on, aligned with the women-first invariant. Venue ops (create + QR + start/stop) is second (it unblocks Bloc 5), aggregate stats third. *Why:* unhandled safety reports are a live gap that directly threatens invariant 5; a nice matches graph is not.
- **Start/stop is a single `SECURITY DEFINER` RPC (`public.set_venue_live`), and `is_live` never moves through a raw `UPDATE`.** Stopping a venue must close *other users'* presence rows, which a founder cannot do under their own RLS, so the operation runs as a definer that self-guards on `private.is_admin()` and does both writes atomically (set `is_live=false` + close all open presence). Venue *creation* stays a normal RLS-gated `INSERT` (`venues_insert_admin`). *Why:* funnelling every liveness change through one guarded path means "stop always empties the room" is structural, and founders never get a broad presence-UPDATE grant.
- **Admin aggregate stats are served only by a counts-only `SECURITY DEFINER` RPC (`public.admin_night_stats`); there is deliberately no admin `SELECT` on `likes`/`matches`.** The function bypasses RLS to count, but returns only per-venue/per-night integers — no ids, no pairs. *Why:* the only way a founder can touch the likes/matches tables is through a function that cannot return anything nominative, making the invariant-3 red line structural rather than a matter of query discipline. (Ephemeral metrics are cleaned nightly, so past nights read 0 for likes/matches/chats — the ephemeral columns are effectively "tonight". Accepted.)
- **The `admins` table is fully locked (RLS on, no policies, no grants); admin-ness is read only through `private.is_admin()`.** Nothing selects `admins` directly; RLS policies and the guarded RPCs call the `SECURITY DEFINER` helper, and the client gate uses a thin `public.am_i_admin()` wrapper. *Why:* the allow-list is a security primitive — the smaller its callable surface, the better. The advisor's `rls_enabled_no_policy` INFO on `admins` is therefore expected, not a gap.
- **The admin UI is English-only internal tooling, outside the FR/EN/ES dictionary.** *Why:* the bilingual dictionary exists for the public, user-facing app; the founder console is internal, and the repo-language rule already mandates English — a second locale set here would be pure overhead.
- **Founder accounts were created directly in `auth.users` via SQL (MCP) with a shared temporary password, not through a signup flow.** The two founder emails were inserted into `auth.users` + `auth.identities` (email provider) and their ids into `public.admins`; the temp password is changed on first use. This account seed is a one-off op, not a migration file. *Why:* the MCP has no admin-create-user tool and there is no public login flow to piggyback on; a direct insert is the pragmatic bootstrap for exactly two internal accounts. Not a pattern for real users — Phase 2 accounts would go through a proper signup/verify flow.
- **The product name changes from Paramour to Amourette (reverses the 2026-06-28 name decision); domain `getamourette.com`.** "Une amourette" = a light, fleeting love affair — it fits the north star (live, discreet, ephemeral, in-person). Due diligence: no dating app of that name on either store, and no software/dating trademark (Class 9/45) — the existing Amourette marks are lingerie (Triumph, Class 25) and perfume (Class 3), different classes. `amourette.com`/`.fr` are taken, hence `getamourette.com`. *Why:* Paramour was blocked/weak; Amourette carries the exact meaning we want. Trade-off accepted: "amourette" is a common French word and an established lingerie brand, so the mark is diluted (weak SEO/ownership) — usable for a dating app but not something we'll ever fully own; a real IP clearance precedes any filing. The Paramour→Amourette rename across UI/docs is a separate chore; DB `bartap-` codename objects stay untouched (per 2026-06-28) unless a migration renames them.
- **Visual identity: dark UI, Bodoni Moda wordmark, a red-derived (monochrome-red) palette.** Dark is the product's ambiance (opened in a bar at night, discreetly), consciously overriding Marwane's Reza-scoped "no dark mode" rule. The logo has three lockups (red / cream / ink) and red is reserved for strong moments; the full palette is derived from a single signature red (`#D01F3C` proposed). Full spec, tokens and open points live in `docs/design.md`; the visual board is `docs/brand/board.html`. *Why:* a dark, editorial, single-red system is coherent with a red wordmark, premium, and ownable in feel even though the *name* is diluted — the DA is where Amourette earns distinctiveness. Three points remain open (red-usage philosophy, exact red, palette confirmation) before tokens are wired into Tailwind.
